> 本文档为 **SafetyZone（Jetson 边缘部署 + Python/PySide6）** 的目标系统设计与实施方案。

> **与既有系统关系**：现场曾运行 **PyQt6 / Windows 版「安全区深度检测」**（双目深度 + 多工位 + PLC）。本仓库为 **Jetson 重构版**，不直接拷贝旧代码，但 **UI 与功能按 [MIGRATION_SPEC.md](MIGRATION_SPEC.md) 做语义等价迁移**；Bootstrap 阶段按 D1 等决策裁剪（先 person + 2D 划区）。
>
> **实施配套文档**：[执行方案](安全区入侵检测系统_执行方案.md) · [验证阶段](validation_phases.md) · [决策记录](decisions.md) · **[功能迁移对照](MIGRATION_SPEC.md)**

## 1. 系统目标与边界

**做什么**:工业产线安全区入侵检测。多路相机实时监视"减速区(SLOW)/停止区(STOP)",当人员或指定物体进入时,经 PLC 联动产线**减速或停机**;并对报警事件录像留证。

**部署形态**:单台 Jetson 作为现场边缘机,直接连相机与 PLC,产线现场独立运行。模型可在现场持续迭代(见 §8 闭环)。

**规模边界**:相机 ≤4 路(Jetson Orin Nano Super 8GB 在此规模下算力/内存充裕;详见 §3)。

**不做什么**:不追求云端依赖,数据不出厂;不承担硬实时安全保证(见 §2)。**Bootstrap 阶段不做**旧版全部特性的一次性平移（深度/权限/训练 Studio 等见 §1.4 与 MIGRATION_SPEC §20）。

### 1.4 与既有系统（PyQt6 版）的关系

| 维度 | 既有系统（迁移源） | 本仓库 SafetyZone（迁移目标） |
|------|-------------------|------------------------------|
| 平台 | Windows + PyQt6 + CPU/GPU 推理 | **Jetson + PySide6 + TensorRT FP16** |
| 判区 | 多边形 + **双目深度距离档** | Bootstrap：**2D 多边形重叠**（`core/zone.py`）；深度后续按需 |
| 检测 | person / object / 圆桶 HSV | **D1：Bootstrap 仅 person** |
| 信号 | `result_code` 0/1/2/3（见 MIGRATION_SPEC §5.3） | `signal` **-1/0/1/2**（见 §6.3）；UI/PLC 需适配层 |
| UI | 多工位 Tab、标定向导、权限、通讯中心 | Sprint B 起按旧版 **§13 布局** 渐进迁移；详表见 **MIGRATION_SPEC §13** |
| 训练 | 文件菜单内 YOLO Studio | **阶段三 `windows_studio`（Win GPU）**，不在 Jetson 运行 UI |

**迁移追踪**：以 MIGRATION_SPEC 功能 ID（**F01–F38**）为单元；执行阶段与裁剪见 [执行方案 §1.2](安全区入侵检测系统_执行方案.md#12-功能迁移对照概要)。

## 2. 安全定位(承重前提)

本系统的所有检测策略、调参、模型验收都建立在以下定位上,需团队对齐:

1. **零漏检不能由软件保证。** 检测模型输出是概率,软件层只能**压低漏检概率、把策略偏向召回**,无法归零。
2. **视觉检测是监督/增强层,不是唯一安全防线。** "人进停止区必须停"的零漏检要求应由**独立硬件安全回路**承担(光幕 / 安全继电器 / 安全 PLC / 急停)。视觉负责更早预警、覆盖硬件够不到的区域、留证据。
3. **目标是召回优先 + 精度有意识(precision-aware)。** 误停有真实代价,且误停过多会逼现场放宽阈值、反而放回漏检风险。正确做法:先把召回顶到目标线,再在不掉召回的前提下压误检。
4. **现场可训练 ⇒ 现场标注质量直接影响安全模型。** 标错/标漏会教模型学会漏检。压住此风险的唯一可靠手段是 §8 的**冻结测试集召回率验收闸**,它是本方案能用于安全场景的前提,非可选项。

## 3. 硬件与运行环境

### 3.1 Jetson Orin Nano Super 8GB(现场边缘机)
- GPU:1024-core Ampere + 32 Tensor Core;CPU:6-core Arm Cortex-A78AE;内存 8GB LPDDR5(CPU/GPU 共享),带宽 102 GB/s(Super 模式)。
- AI 性能:67 TOPS(INT8);系统:JetPack 6.x(Ubuntu/L4T,arm64)。
- 功耗模式:7/15/25W + MAXN Super。
- **关键约束(影响设计)**:**无硬件视频编码器(NVENC)、无 DLA**;仅有硬件解码器(NVDEC)。→ 录像方案必须规避连续硬编(见 §6.5)。

### 3.2 Windows GPU 工作台(模型作者侧)
- 现场一台带 NVIDIA GPU 的 Windows 11 机,由调试人员操作,承担**阶段三**标注 + 训练 + 导出 + 下发(`windows_studio`)。
- **与 Jetson 运行 UI 分工**:Windows 侧不负责产线监视与划区;日常运行、Bootstrap 集成验证均在 **Jetson PySide6 运行 UI** 完成。

### 3.3 网络与远程开发
- Windows 与 Jetson 同局域网,现场以**直连网线 + 静态 IP** 连接。数据全程不出厂。
- 研发通过 **SSH / Cursor Remote** 在 Jetson 上开发 Bootstrap（export、TRT、运行 UI）；细则见 `validation_phases.md` §1.6、`decisions.md` D12。
- 运行 UI 需 Jetson 本地显示器或 VNC;纯 SSH 无 DISPLAY 时先验证 TRT/CLI。

```
        ┌──────────────────── 现场局域网(直连网线) ─────────────────────┐
        │                                                                 │
 ┌──────┴───────────────┐                          ┌────────────────────┴──────┐
 │  Windows 工作台        │                          │   Jetson 现场机             │
 │  (调试人员)            │  难case 上传 ◀───────────┼── 实时检测 → PLC 联动        │
 │  复核标注 → 微调训练    │                          │   → 报警录像                 │
 │  → 导出 ONNX           │──── ONNX 下发 ──────────▶│   → 收 ONNX → 编 engine      │
 │  → 版本归档            │                          │   → 召回率验收 → 热切换/回滚  │
 └────────────────────────┘                          └─────────────────────────────┘
```

### 4.2 分层架构(平台无关核心 + 平台适配器)

```
┌───────────────────────────────────────────────────────────┐
│ 运行 UI 层 (PySide6)        监视网格 / 划区 / 示教 / 配置     │
├───────────────────────────────────────────────────────────┤
│ 编排层 (app)               线程/进程编排、门控、事件分发      │
├───────────────────────────────────────────────────────────┤
│ 适配器层 (平台相关)                                          │
│   相机(V4L2/Orbbec) · 推理(TensorRT) · PLC(snap7) · 录像     │
├───────────────────────────────────────────────────────────┤
│ 核心层 (core, 平台无关, 纯 Python+numpy)                     │
│   判区几何 · 入侵状态机 · 后处理/NMS/过滤 · 跟踪 · 异常差分    │
│   · 配置模型                                                 │

```

设计原则:**核心层不依赖任何硬件/平台**,可在任意机器开发与单测;适配器层封装所有平台细节。这让逻辑正确性与硬件解耦,便于测试和移植。

| 层 | 选型 |

## 5. 检测流水线

### 5.1 主流程(单工位)

```
取帧 → 预处理(Letterbox) → TensorRT 推理 → 后处理(解析/NMS/过滤/跟踪)
   → 判区(锚点∈多边形 或 重叠面积比) → 入侵状态机(防抖) → 信号 → PLC + 录像
```

- **同相机多工位只推理一次**,推理结果分发给该相机下各工位独立做后处理与判区。
- **多路降频**:≥2 路时检测/预览帧率上限下调,保证 4 路下总负载可控(目标检测帧率本就不需很高)。

### 5.2 三种检测模式(按工位配置)

| 模式 | 是否用模型 | 适用 | 锚点 |
|------|-----------|------|------|
| **person**(主) | 是 | 人员入侵(核心安全功能) | 框底边中点 (cx, y2) |
| **object** | 是 | COCO 类物体(车/料箱等) | 框中心 (cx, cy) |
| **anomaly** | 否 | 通用模型无法覆盖的自定义工件/物料 | 差分 blob 底边中点 |

- person / object 共用同一推理+后处理流水线,仅类别过滤与锚点不同。
- **anomaly 模式**:示教阶段采集各停区背景参考帧;运行时对停区 ROI 做背景差分(灰度 AbsDiff → 阈值 → 形态学 → 轮廓 blob),映射回全帧判区。**不依赖模型**,作为"通用数据集无法覆盖的目标"的兜底手段。

### 5.3 召回优先调参(固化在保守侧)

旋钮分两组管理:**召回组定死保守、不轻易动;精度组是"不掉召回前提下压误检"的工作区。**

| 旋钮 | 偏召回方向 | 组 |
|------|-----------|----|
| 推理 `conf` 置信度 | 调低 | 召回 |
| 入侵进入帧数 EnterFrames | 调小(更快触发) | 召回 |
| 入侵退出帧数 ExitFrames | 调大(更慢解除) | 召回 |
| 漏检保持 HoldMs | 调大(漏检帧沿用上帧) | 召回 |
| 重叠阈值 MinOverlap | 调低(沾边即判) | 召回 |
| 安全区多边形 | 外扩 margin | 召回 |
| 尺寸门控 / 框校验器 | 在不掉召回前提下收紧 | 精度 |
| NMS / 场景类排除 | 压重复框/家具误检 | 精度 |

## 6. 子系统设计

### 6.1 相机采集
- 统一抽象 `CameraStream`:`start/stop`、`get_frame()`、可选 `get_depth()`、连接状态事件。
- USB:V4L2 / GStreamer 取流(枚举 `/dev/video*`);双目并排帧按宽高比识别并拆分左右 ROI,只输出单目。
- Orbbec:pyorbbecsdk 取 RGB(若主链路不需深度,先只接 RGB,降低复杂度)。
- 每路只保留**最新帧**、主动丢中间帧;**帧入缓冲必须 `.copy()`**(numpy 切片是 view,不复制会被采集线程复写污染)。
- 自动重连:断线后重试(Watchdog),区别于"单次 open 后死循环读"。

### 6.2 推理(TensorRT FP16)
- 模型导出:`yolo export model=yolov8s.pt format=onnx imgsz=640 opset=18 simplify=True dynamic=False`。
- 引擎编译(**必须在 Jetson 本机**,engine 与 GPU+TensorRT 版本绑定不可跨机):
  `trtexec --onnx=model.onnx --saveEngine=model.engine --fp16`。
- 后端 `trt_backend`:Letterbox 预处理 → TensorRT 推理 → 输出解析。**FP16 为默认**(精度几乎无损);**INT8 不作默认**(实测提升有限且掉精度,仅在内存/功耗紧张时启用,且需现场校准集 + 精度门槛)。
- 支持**运行时热切换 engine**(配合 §8 模型更新):加载新 engine → warmup → 原子切换,不在单帧检测中途切。

### 6.3 判区与入侵状态机

**判区**:
- 安全区为多边形(slow / stop),定义于参考分辨率,判区时缩放到当前帧。
- 命中规则:锚点落在多边形内即命中(快路径);否则计算检测框与多边形 mask 的重叠面积比 ≥ MinOverlap。
- 嵌套优先级:内 STOP 区 > 外 SLOW 区 > 无。

**入侵状态机(帧级防抖,slow/stop 双通道独立)**:
- 进入:连续 EnterFrames 帧有目标 → active。
- 退出:连续 ExitFrames 帧无目标 → inactive。
- **信号语义(本系统定义)**:

| 值 | 含义 |
|----|------|
| `2` | STOP 确认(停线) |
| `1` | SLOW 确认(减速) |
| `0` | 有原始入侵但未确认(过渡/预警) |
| `-1` | 区内无目标 **或** 模型/相机故障 |

### 6.4 PLC 通信(python-snap7)
- 单连接 + 串行化 Gateway(避免并发断连);自带 `auto_reconnect`/重试/断线回调。
- **Bootstrap 仿真模式**(见 decisions D11):未接 PLC 真机时,UI 展示拟写入的 INT16 结果,不发起 snap7 连接;用于 Jetson 运行 UI 首验。
- 两种协议模式(按现场 PLC 程序选):
  - **block 模式**:每工位读固定字节块 @ offset;块内字段约定 心跳/触发/区域切换/结果/距离 的偏移。上升沿 + TriggerId 触发单次检测。
  - **command 模式**:读变量表 → 命令字解析 → 激活工位集合 → 持续门控检测;`PlcResultWriter` 按规则持续回写。
- **结果回写(本系统定义 INT16 语义)**:`0` 安全 / `1` SLOW / `2` STOP / `-1` 故障(无结果/stale)/ `3` 命令字>0 但无匹配工位。写后读回校验,失败重试。
- **看门狗**:整轮轮询超时 或 连续读失败达阈值 → 断连;断连可配置保持已激活工位(UI 标 OFFLINE·HOLD)。
- **进程隔离**:PLC 通信跑在**独立进程(multiprocessing)**,经队列与主进程换信号,避开 GIL 与 Qt 事件循环抖动;阈值在 Jetson 实测周期抖动后整定。

### 6.5 录像(无 NVENC)
- 每工位环形缓冲**预录**若干秒;报警边沿(SLOW/STOP)触发后续录若干秒 + 快照。
- 编码后端(默认保守,二选一):
  - **快照 + 短片段**(推荐默认):报警存关键帧 + 少量帧,规避连续编码。
  - GStreamer `x264enc` 软编:限分辨率/码率,独立低优先级线程,避免与推理抢 CPU。
- 保留策略:按天数 / 最小空闲磁盘 / 最大占用 清理。
- **注意**:若现场需长时间连续录像(非短片段),无 NVENC 的 Orin Nano 会吃紧,届时需重新评估硬件(Orin NX 有 NVENC)。

### 6.6 配置体系
- 单一 `config.json`:`stations[]`(工位:绑定相机+区域+检测模式+参数组)、`cameras[]`、`param_groups[]`(slow/stop 多边形+参考分辨率+阈值)、`plc`(ip/rack/slot/db/协议模式/变量表/看门狗)。
- 加载健壮性:损坏回退备份;**原子写**(tmp→replace)+ 时间戳轮换备份 + 审计日志。
- 加载时归一化:补全双目 ROI、校验 SN/工位绑定唯一性、为多工位克隆独立参数组模板。

### 6.7 运行 UI(PySide6) — 部署于 Jetson

- **Bootstrap 阶段即在 Jetson 部署完整运行 UI**,用 COCO 预训练 stock 模型 + 录制视频/USB 完成集成验证(非仅 CLI 测 YOLO)。细则见 `validation_phases.md`。
- **布局基准**：沿用既有「安全区深度检测」主窗口结构（菜单 + 工具栏 + 左侧相机列表 + 工位 Tab + 日志 + 状态栏），见 **MIGRATION_SPEC §13.1–13.3**；Sprint B 先交付单工位最小集，多 Tab / 总览 / 标定向导分 Sprint 递进。
- 监视网格(2×2 够用)、运行状态/报警指示、安全区划区编辑、PLC 配置窗。(anomaly 示教按 D1 可省略)
- 界面须能标识 **「STOCK · 集成测试 · 未过场内验收」**,避免与 Production 模型混淆。
- 刷新分频:视频 ~15FPS / 仪表盘 ~1FPS / 状态栏 ~2FPS;缩放防抖。
- 纪律:**检测/IO 不在 UI 线程**;后台算、信号槽 marshal 回 UI(Qt 信号槽替代直接改控件)。
- **帧推送**：等价于旧版 `FrameBridge`（MIGRATION_SPEC §5.4）；最小字段含 `station_id`、`signal`、检测框 overlay、`fps`、PLC 拟写入值（仿真模式）。

## 7. 并发模型(GIL-aware)

| 任务 | 载体 | 说明 |
|------|------|------|
| 相机采集 | 每路一线程 | cv2 取帧/处理在 C 层释放 GIL,可真并行 |
| 推理调度 | 全局一线程 | GPU 串行;TensorRT 调用释放 GIL |
| 后处理/判区/FSM | 线程内 | numpy/cv2 释放 GIL;纯 Python 热循环受 GIL,保持轻量 |
| PLC 通信 | **独立进程** | 隔离 GIL/Qt 抖动,保时序确定性 |
| 录像软编 | 独立低优先级线程 | 避免与推理抢 CPU |
| UI | 主线程(Qt 事件循环) | 仅展示 |

要点:**图像/推理重活在 C 扩展里执行、会释放 GIL,故 GIL 不是主要瓶颈;真正约束是 6 核 CPU 的总量争抢**(前处理 + anomaly 差分 + 录像软编都在 CPU)。4 路 + 低帧率下大概率充裕,需在 §10 阶段二压测确认。

## 8. 现场训练闭环

### 8.1 数据流(端到端)

```
Jetson outbox(难case 原图+预测) ──rsync──▶ Windows 工作台
   Windows: 预标注 → 复核 → 入训练集 → 微调 → 导出 ONNX
ONNX ──rsync──▶ Jetson inbox
   Jetson: trtexec 编 engine → 冻结测试集召回率验收 → 通过热切换 / 否则拒绝保留旧版
```

### 8.2 Windows 工作台(PySide6 闭环工具)

操作者 = 调试人员;要求**单一入口、低使用成本**。**不套通用标注器**(CVAT/Label Studio 对本闭环偏重);因有预标注垫底,人只做"复核/纠正框",故自建集成工具使用成本最低。

向导式四步:
1. **拉取**:从 Jetson outbox 拉回难 case(低置信/近区/疑似漏检的帧)。
2. **复核**(UX 重心):预标注已画好框,界面只需 `确认 / 拖动改框 / 删误框 / 补漏框` + 翻页 + 类别下拉;**引导"宁可多标、勿漏标"**(漏标对安全系统更危险),疑似漏检帧高亮。
3. **训练**:并入训练集,从上版 checkpoint 微调(`yolo train`,Windows CUDA);参数有安全默认、对调试人员可见可改;显示进度。
4. **下发**:导出 ONNX → 经局域网送 Jetson inbox → 工具内显示 Jetson 验收结果。

模块:`ingest` / `preannotate` / `review_ui` / `dataset`(训练/测试集物理隔离 + 重叠校验) / `train` / `export_send` / `registry`(版本档案+回滚)。

### 8.3 Jetson 模型接收侧(两段式后半段)

> Windows 只传 ONNX(engine 不可跨机)。Jetson 负责把 ONNX 变成可上线 engine 并把住验收闸。

- `receiver`:监听 inbox,收到 ONNX 触发。
- `build_engine`:`trtexec --fp16` 编译候选 engine。
- `acceptance`:用**冻结测试集**跑候选 engine 算**召回率**(误检率作参考);**召回率 < 阈值 → 拒绝**(保留现行 engine + 告警),通过 → 热切换。
  - 验收必须在 **Jetson + FP16 engine 最终形态**上做(Windows 的 PyTorch/ONNX 精度 ≠ Jetson FP16 实际精度)。
- `hotswap`:加载 → warmup → 原子切换;保留上版,支持 `rollback`。

### 8.4 冻结测试集纪律(比代码更重要)
- 测试集针对**场内微调模型**的上线验收;**COCO 预训练 stock 模型**用于 Bootstrap 集成验证时**不要求**建立场内冻结集。
- 测试集自建立第一天**锁死,永不进训练集**;混入即令验收失效(模型见过答案)。
- 测试集带标注,由调试团队一次性认真标注、冻结,存于 Jetson 供验收。
- 每次训练前**校验训练集与测试集无帧重叠**,有则拒绝训练。
- 靠**流程纪律**保证,出问题不报错、只悄悄让安全闸失效——定为团队规矩。

### 8.5 模型策略
- **冷启动(Bootstrap)**:YOLOv8s **COCO 预训练** → Jetson 导出 ONNX、编 FP16 engine → **运行 UI** 上用视频/USB 验证全流水线。COCO 仅证明集成可行,**不能**替代场内安全验收。
- **迭代(Production)**:场内难 case → `windows_studio` 复核微调 → 冻结测试集召回验收 → 热切换;几百到低千帧针对性样本即可缩小域差。
- **新微调模型不自动上线**:必过 §8.3 验收闸 + 版本化可回滚。stock 模型可用于集成联调(含 PLC 仿真/真机),但须标「未过场内验收」。

## 11. 关键纪律 / 护栏

1. `.engine` 只在 Jetson 编,不跨机拷;Windows 只出 ONNX。
2. **微调模型**不自动上线;必过场内冻结测试集召回率验收(stock 模型 Bootstrap 除外,但须标集成测试)。
3. 模型版本化 + 一键回滚。
4. 场内冻结测试集永不进训练,训练前校验无重叠。
5. 召回优先;视觉为监督层,零漏检靠独立硬件安全回路。
6. OpenCV 必须带 GStreamer;帧入缓冲必须 `.copy()`;PLC 进程隔离。
7. 内存只认完整应用运行下的实测;FP16 默认,INT8 是最后手段。
8. 检测/IO 不在 UI 线程。
9. **勿将 COCO Bootstrap 集成验证当作正式安全验收。**
10. **运行 UI 在 Jetson;训练 UI 在 Windows studio** — 不混用。
11. **UI/功能迁移以 MIGRATION_SPEC F ID 追踪**；旧 `result_code` 与新 `signal` 须经适配层，禁止混用数值语义。

## 12. 待确认 / 我做的判断(你来校验)

- **三种检测模式是否都要**:person 是核心;object/anomaly 是按需。若现场只检人,可砍掉 object/anomaly,系统更简（**D1 已定仅 person**；旧版 object/圆桶见 MIGRATION_SPEC F12/F13，Jetson 不迁）。
- **Bootstrap 是否以 Jetson 完整运行 UI 为验收主体**:已定 **是**;见 validation_phases.md。
- **UI 迁移是否对齐旧版布局**:已定 **是（渐进）**；Sprint B 单工位最小集 → 多工位/标定/权限见 MIGRATION_SPEC §20 Phase B–D。
- **Bootstrap 判区用 2D 还是深度**:建议 **先 2D 多边形**（与现 pipeline 一致）；双目深度 + 标定向导为 Phase C，见执行方案 §1.2。
- **stock 模型是否允许 PLC 联调(含仿真)**:已定 **允许仿真首验**;真机联调须标集成测试(D11)。
- **召回率验收阈值**:需你与现场基于**场内**冻结测试集和可接受漏检率共同定一个具体数,我无法替定。
- **标注 UI 自建 vs 套通用底座**:基于"调试人员 + 简化成本 + 有预标注"判断自建更省;若预期标注复杂化(多类/分割/协作),可改用通用底座作后端。
- **Orbbec 深度是否进主链路**:若只用 RGB,复杂度显著下降。
- **录像是否需长时间连续**:若需要,无 NVENC 的 Orin Nano 会吃紧,可能指向 Orin NX。
- **PLC 型号**:S7-1200/1500 若关闭 PUT/GET,需走 python-snap7 的 S7CommPlus 路径。
- **A100 远端训练**:当前 **不采用**;训练仅在调试人员 Win GPU(D8)。
- **Jetson Remote 开发**:已定 **是**;Sprint 1.2 起板上主验证(D12)。
